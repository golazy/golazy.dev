type PostsController struct {
  controllers.BaseController
}

func New(ctx context.Context) (
  *PostsController, error,
) {
  base, err := controllers.NewBaseController(
    ctx, "posts",
  )
  if err != nil {
    return nil, err
  }

  posts, _ := postservice.FromContext(ctx)
  return &PostsController{
    BaseController: base,
    posts: posts,
  }, nil
}

func (c *PostsController) Index(
  w http.ResponseWriter,
  r *http.Request,
) error {
  c.Set("posts", c.posts.List())
  return c.Render("index")
}

01 / Why GoLazy

Small framework.
Big momentum.

Spend less time wiring the same pieces together and more time shaping the application. GoLazy gives every concern a predictable home while keeping the standard library close.

01

Convention earns speed.

Controllers and actions map naturally to view folders and template names. The obvious path is already paved.

02

One binary goes anywhere.

Templates, public assets, and content are embedded. Build once, deploy a single executable.

03

Dependencies stay visible.

Application context carries shared services. Controllers receive one thing and begin ready to work.

02 / Everything in its place

A full-stack shape
without the full-stack weight.

app/controllers

Controllers handle requests.

Actions set view data, coordinate services, and render a response through a shared application controller.

app/views

Templates stay close to intent.

Each controller owns a view directory. Rendered content flows through layouts for a consistent shell.

app/services

Services keep work focused.

Reusable application behavior lives outside HTTP concerns and is initialized once in the application context.

golazy.dev

The framework stays extractable.

Routing, controller primitives, rendering, and fallbacks live in small packages published as the standalone GoLazy framework.
